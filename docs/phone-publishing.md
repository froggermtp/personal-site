# Publishing notes from your phone

This sets up two Android home-screen shortcuts, built with the
[HTTP Shortcuts](https://http-shortcuts.rmy.ch/) app
(`ch.rmy.android.http_shortcuts`, open source, F-Droid / Play Store), that
publish to the site's "Notes" microblog directly from your phone:

- **Post Note** — a text-only note. Share text into it from any app, or tap
  it and type.
- **Post Photo Note** — a note with a photo. Share an image into it from
  your gallery or any share sheet; it asks for a caption and alt text.

Both shortcuts commit straight to `main` on GitHub via the GitHub Contents
API. GitHub Actions then builds the site and Netlify deploys it — the same
pipeline as any other push to `main`. There is no server of your own to run.

This guide was written against the app's public docs
(<https://http-shortcuts.rmy.ch/scripting>, `/variables`, `/shortcuts`,
`/import-export`, `/advanced`, `/execution-flow`) and the app's actual
open-source data model
(<https://github.com/Waboodoo/HTTP-Shortcuts>, `import_export/models/`).
Everything below is grounded in one of those two sources except where
explicitly flagged as untested. **One step — reading a shared photo's raw
bytes as base64 inside a script — relies on app behavior that isn't fully
documented and must be verified on your device.** See
["The one real unknown"](#the-one-real-unknown-reading-photo-bytes) below.

---

## 1. Create a fine-grained GitHub PAT

1. On github.com: **Settings → Developer settings → Personal access tokens
   → Fine-grained tokens → Generate new token**.
2. **Repository access**: "Only select repositories" → `froggermtp/personal-site`.
   Do not grant access to any other repository.
3. **Permissions → Repository permissions**: set **Contents = Read and
   write**. Leave every other permission at "No access" — the shortcuts only
   need to read/write file contents on this one repo.
4. **Expiration**: 90 days is a reasonable default. Put a reminder to
   rotate it (generate a new token, paste it into the `github_token`
   variable in the app, delete the old token on GitHub).
5. Copy the token (`github_pat_...`). You'll paste it into the app in the
   next step and never need to type it again until it's rotated.

**Security notes**

- The token lives in exactly one place: the `github_token` variable inside
  HTTP Shortcuts on your phone. It is not committed anywhere, and the JSON
  file in this repo (`docs/http-shortcuts-notes.json`) ships with that
  variable's value left empty.
- The repo is **public**, so never paste the token into a commit, a note
  body, an issue, or anywhere else that ends up in git history or on the
  site.
- If your phone is lost or stolen, revoke the token immediately from
  **Settings → Developer settings → Fine-grained tokens** on github.com —
  this takes effect instantly and doesn't require phone access.
- Because the token is scoped to Contents:read/write on a single public
  repo, the worst case of a leak is someone pushing junk commits to this
  repo, which you can revert — it can't touch any other repo or account
  setting.

## 2. Import the shortcuts

1. Get `docs/http-shortcuts-notes.json` onto your phone (e.g. open the raw
   file on GitHub in your phone's browser and save/share it, or sync it via
   whatever you use to move files to the phone).
2. Open HTTP Shortcuts → the three-dot menu on the main screen → **Import /
   Export**.
3. Choose **Import from file** (there's also **Import from URL** if you'd
   rather host the raw GitHub URL and import directly from there) and pick
   the JSON file.
4. You should end up with a new **Notes** category containing two
   shortcuts, **Post Note** and **Post Photo Note**, plus three new global
   variables: `github_token`, `New Note` (key `note_text`), and `Alt Text`
   (key `note_alt_text`).

   This file is a plain, uncompressed JSON export (not a `.zip`). That's a
   supported import format — the app's importer tries to read the file as a
   zip first and, if that fails, falls back to parsing it as plain JSON
   directly (confirmed from the app's `Importer.kt` source, which is the
   `shortcuts.json` format the app itself writes when you choose "Legacy
   JSON" as the export format on the Import/Export screen). It is **not**
   auto-generated by the app — it was hand-built to match the app's
   documented schema. If import fails on your device (schema drift between
   app versions is the most likely cause), use the
   [manual setup instructions](#manual-setup-fallback) below instead; treat
   them as the source of truth if the two ever disagree.
5. Open the global variable **GitHub Token** (key `github_token`) and paste
   your PAT into its value. Save. This is the only manual configuration
   step after import.

## 3. Shortcut A — "Post Note" (text only)

**Trigger it two ways:**
- Select text in any app → Share → **Post Note**. The shared text becomes
  the note body immediately, no typing required.
- Tap the shortcut directly (from its home-screen icon, or from inside the
  app). Since no text was shared, you get a text-entry dialog ("What's on
  your mind?") to type the note.

Both paths work because `note_text` is a global **Text Input** variable
with "Allow Receiving Value from Share Dialog" enabled
(`isShareText: true` in the JSON). When the shortcut asks for that
variable's value — which it does explicitly inside its script, not
implicitly — the app either uses the shared text (if the shortcut was
invoked via a share) or shows the input dialog (if not). This early,
explicit resolution is a documented pattern: the execution-flow docs note
that a variable can be "resolved before [the normal resolution step],
e.g. if you ask for their value via Scripting in step 5 or 6" — step 6
being exactly the "Run before Execution" script this shortcut uses.

**What the script does** (all of it — this shortcut has no separately
configured URL/method/body; it's a "Scripting"-type shortcut that is
*only* a script):

1. Reads `note_text`.
2. Builds the slug (`YYYYMMDD-HHMM`, device-local time) and an ISO 8601
   timestamp with the device's local UTC offset.
3. Assembles the markdown: a `---`-fenced frontmatter block with just
   `date`, then the note body.
4. Base64-encodes it and `PUT`s it to
   `https://api.github.com/repos/froggermtp/personal-site/contents/src/views/notes/<slug>.md`
   with headers `Authorization: Bearer {{github_token}}`,
   `Accept: application/vnd.github+json`, `X-GitHub-Api-Version: 2022-11-28`,
   and a JSON body `{"message": ..., "content": <base64>, "branch": "main"}`
   — using the app's `sendHttpRequest()` scripting function, which is
   synchronous (no `await` needed) and returns
   `{status: "success"|"httpError"|"networkError", response: {statusCode,
   body, headers, cookies}, networkError}`.

**Success**: a toast, "Note published: `<slug>`".
**Failure**: a dialog titled "Post Note" showing either the HTTP status
code and response body (bad token, permissions, rate limit, etc.) or a
network-error message (no connectivity).

Nothing is shown by the app's own default response handling — this
shortcut type has none — all feedback comes from the script's own
`showToast()`/`showDialog()` calls.

## 4. Shortcut B — "Post Photo Note"

**Trigger it:** from your gallery or any app, Share an image → **Post
Photo Note**. It then prompts for the caption (reusing the same "What's on
your mind?" dialog as Post Note) and then for alt text ("Describe the
photo for accessibility"). You can also tap the shortcut directly, which
opens a file picker to choose an image (see
[design decisions](#design-decisions-for-shortcut-b) below for why).

**What happens, in order:**

1. The shared image becomes available to the script as `selectedFiles[0]`.
2. It prompts for `note_text` (caption) and `note_alt_text`. Cancelling
   either aborts with a toast, nothing is sent.
3. It reads the image's raw bytes and base64-encodes them (see the
   [unknown](#the-one-real-unknown-reading-photo-bytes) below).
4. It builds the same slug/date as Shortcut A, then `PUT`s the **image
   first** to
   `.../contents/src/assets/images/notes/<slug>.jpg` — this ordering is
   required, because the site's build reads the image file at build time
   and fails if the markdown references one that doesn't exist yet on
   `main`.
5. Only if that succeeds, it `PUT`s the markdown (frontmatter now includes
   `image: notes/<slug>.jpg` and `imageAlt: "..."`) to
   `.../contents/src/views/notes/<slug>.md`.
6. Reports success or failure via toast/dialog directly from the script.

If the image upload fails, it stops there (no orphaned markdown pointing
at a missing image). If the image succeeds but the markdown commit fails,
you get a dialog saying so — the image is already on `main` at that point,
so you'd need to either retry (which will pick a new, later slug and
re-upload the image under a new name) or add the markdown file by hand on
GitHub referencing the already-uploaded image path.

### Design decisions for Shortcut B

**Why this shortcut has to be a Regular/HTTP shortcut (`executionType:
"app"`), unlike Post Note.** Only Regular shortcuts have a "Request
Body / Parameters" section where you can add a parameter of type "Single
File" — and that's what makes Android's share sheet (and, on Android 11+,
the faster Direct Share sheet) offer this shortcut as a target for images,
and what makes `selectedFiles` populate. The docs are explicit: *"If your
shortcut makes use of file parameters or uses the content of a file as the
request body then you can access information about these files using the
`selectedFiles` array."* A pure "Scripting" shortcut (like Post Note) has
no such section, so it can't register as a file share target.

**Why the shortcut's own declared request is a dead end, and what fills
in for it.** Registering as a file share target requires Request Body Type
= "File (Picker)" or "Parameters (form-data)" — both of which mean the
shortcut's *actual* declared request body is either the raw file or a
multipart form, not the hand-built JSON `{"message", "content", "branch"}`
that GitHub's Contents API needs. There's no documented way to have one
shortcut be both a file share target *and* send a custom JSON body as its
declared request. So this shortcut's declared request (`POST` to
`https://api.github.com/repos/froggermtp/personal-site`, a bare repo-info
endpoint that rejects `POST`) is a placeholder that's never meant to
actually matter — the real work (both GitHub API calls) happens inside
the "Run before Execution" script via `sendHttpRequest()`, which is
confirmed synchronous, and the script calls `abort()` at the end to skip
the placeholder request. `abort()`'s scripting-context restrictions aren't
fully documented (only `abortAndTreatAsFailure()` is explicitly scoped to
the "Run on Success" block), so as a safety net the placeholder request is
configured to be harmless even if it *does* fire — it would just get
rejected by GitHub with no side effects, and the shortcut's response
handling is set to show nothing (`successOutput`/`failureOutput: "none"`)
so a stray rejection wouldn't confuse you with a misleading toast on top
of the real one the script already showed.

**Why "File (Picker)" as the fallback upload type.** The `fileUploadType`
field controls what happens if you trigger the shortcut *without* sharing
a file (e.g. tapping its icon directly) — options are `file_picker`,
`file_picker_multi`, `camera`, `stored_file` (a file in a mounted
directory), or `static_value`. `file_picker` was chosen so tapping the
icon directly still works (opens your gallery/file picker) as a fallback
to sharing. Sharing an image in is the documented, expected path (it
carries the file in regardless of this setting); tap-to-pick is a
convenience on top.

**Images are always committed as `.jpg`**, per the site's content model.
If you share a non-JPEG image, its bytes still get committed under a
`.jpg` name — prefer sharing JPEGs (most gallery/camera apps default to
this).

### The one real unknown: reading photo bytes

`selectedFiles` and reading file content from a script are genuinely
documented (verbatim, from the scripting docs):

> "The `getDirectory` function can also be used to read the contents of
> files that have been selected or shared, i.e., the files listed in
> `selectedFiles`. Instead of passing the name or ID of a mounted
> directory, simply omit the argument. You can then call the `readFile()`
> function on the returned object, using either the file name or ID as the
> first argument."

And `readFile()` does take an optional second "encoding" argument:

> "As an optional second argument, you can pass the encoding that should
> be used to read the file, which defaults to UTF-8."

**What's not documented anywhere in the public docs is the list of valid
values for that encoding argument.** No example shows `readFile()` used
with an image or any binary file, and none of the base64 examples on the
page combine it with `readFile()`. If the encoding argument only accepts
text encodings (UTF-8, etc.) and not something like `"base64"`, reading a
binary JPEG through it would corrupt the bytes — UTF-8 decoding is lossy
for arbitrary binary data.

The script in `docs/http-shortcuts-notes.json` calls
`getDirectory().readFile(file.id, 'base64')` — a reasonable,
documented-shaped guess (it's a very common convention for exactly this
situation in similar scripting APIs) but **not confirmed**. **Test this
before trusting it with a real photo:**

1. Share a small, distinctive test image (e.g. a photo of a printed page
   of random text, or anything where corruption is visually obvious) to
   **Post Photo Note**.
2. After it reports success, open the committed file on GitHub at
   `src/assets/images/notes/<slug>.jpg` and check whether GitHub renders it
   as a valid image. A corrupted encoding will produce a file GitHub can't
   preview, or a garbled/broken image.
3. If it's broken, the practical next step is a small Netlify Function
   that accepts a raw image body plus a shared-secret header and commits
   it to GitHub server-side — sidestepping the phone-side base64 question
   entirely, since server code has unrestricted binary handling. This is
   **not implemented** as part of this setup; it's a fallback to build
   later if step 2 shows corruption. If you get there, Shortcut B's
   Request Body Type would stay "File (Picker)" (simpler than form-data,
   since there's no longer a need to also read the bytes in script) and
   its declared request would `POST` the raw file straight to the
   function's URL with the shared secret as a header — no scripting
   needed on the phone at all for the image half.

If it turns out `readFile()` needs a different argument (or none, with a
separate step to get bytes), the fix is confined to that one line of the
script — `docs/http-shortcuts-notes.json` can be re-imported after editing
it on-device in the app's own shortcut editor (Scripting tab → "Run before
Execution").

## 5. How publishing works end-to-end

1. The shortcut's `PUT` request creates a commit directly on `main` via
   GitHub's Contents API (this is what a `PUT .../contents/<path>` call
   does — create-or-update a file in one commit, no PR, no local git
   needed).
2. That push triggers the repo's GitHub Actions workflow (same as any
   other push to `main`).
3. Actions builds the site; Netlify picks up the build and deploys it.
4. Total latency from tapping the shortcut to the note being live is
   whatever the Actions build takes — typically a few minutes. There's no
   way to speed this up from the phone; it's the same pipeline as pushing
   from a laptop.

## 6. Troubleshooting

- **401 Unauthorized** in the failure dialog: the token is missing,
  wrong, expired, or was revoked. Generate a new fine-grained PAT (Step 1)
  and paste it into the `github_token` variable (edit the variable in HTTP
  Shortcuts, paste, save).
- **409 Conflict or 422 Unprocessable Entity**: a file already exists at
  that path — i.e., a note was already published in the same
  device-local minute (the slug is `YYYYMMDD-HHMM`, one-minute
  granularity). Wait a minute and retry, or edit the shortcut's script to
  bump the slug manually for that one run.
- **Commit shows up on GitHub but the note isn't live on the site**: the
  publish step (commit) succeeded but the build/deploy step hasn't
  finished or failed. Check the **Actions** tab on
  `github.com/froggermtp/personal-site` for the workflow run triggered by
  your commit — it'll show build errors if any (e.g. a markdown/frontmatter
  problem, or for photo notes, a missing image if something went wrong
  with the ordering).
- **Photo note: image committed but markdown commit failed**: see the
  note at the end of [Section 4](#4-shortcut-b--post-photo-note) — the
  image is already live on `main`; you'll need to add the markdown file
  by hand (or retry, which uploads a new image under a new slug rather
  than reusing the orphaned one).

## Manual setup fallback

If importing `docs/http-shortcuts-notes.json` fails or behaves
unexpectedly on your device, here's how to build both shortcuts by hand.
Field names below match what's in the JSON file and the app's own schema,
which should line up with the on-device editor's labels.

### Variables (create these first: main menu → Variables → add)

| Key | Type | Settings |
|---|---|---|
| `github_token` | Static Variable (Constant) | Value: leave empty for now. Mark it as a secret value so it's masked in the editor. |
| `note_text` | Text Input | Title "New Note", message "What's on your mind?", multi-line on. In the variable's advanced settings, enable "Allow Receiving Value from Share Dialog" and set it to use the shared **text**. |
| `note_alt_text` | Text Input | Title "Alt Text", message "Describe the photo for accessibility". Leave share-receiving off. |

After creating `github_token`, edit it and paste your PAT into its value.

### Shortcut A: "Post Note"

1. Create a new shortcut, type **Scripting**.
2. Name it "Post Note".
3. In its script editor (this is the shortcut's *only* code — Scripting
   shortcuts have no separate URL/method/body fields), paste:

```js
function padNum(n) { return (n < 10 ? '0' : '') + n; }
function buildSlug(now) {
  return '' + now.getFullYear() + padNum(now.getMonth() + 1) + padNum(now.getDate())
    + '-' + padNum(now.getHours()) + padNum(now.getMinutes());
}
function buildIsoLocal(now) {
  var offsetMinutes = -now.getTimezoneOffset();
  var sign = offsetMinutes >= 0 ? '+' : '-';
  var abs = Math.abs(offsetMinutes);
  return '' + now.getFullYear() + '-' + padNum(now.getMonth() + 1) + '-' + padNum(now.getDate())
    + 'T' + padNum(now.getHours()) + ':' + padNum(now.getMinutes()) + ':' + padNum(now.getSeconds())
    + sign + padNum(Math.floor(abs / 60)) + ':' + padNum(abs % 60);
}

var noteText = getVariable('note_text');
if (noteText === null || noteText.trim() === '') {
  showToast('Post Note cancelled.');
  abort();
}

var now = new Date();
var slug = buildSlug(now);
var isoLocal = buildIsoLocal(now);

var markdown = '---\n' + 'date: ' + isoLocal + '\n' + '---\n' + noteText.trim() + '\n';
var path = 'src/views/notes/' + slug + '.md';
var url = 'https://api.github.com/repos/froggermtp/personal-site/contents/' + path;

var result = sendHttpRequest(url, {
  method: 'PUT',
  headers: {
    'Authorization': 'Bearer ' + getVariable('github_token'),
    'Accept': 'application/vnd.github+json',
    'X-GitHub-Api-Version': '2022-11-28',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    message: 'Add note ' + slug,
    content: base64encode(markdown),
    branch: 'main'
  })
});

if (result.status === 'success') {
  showToast('Note published: ' + slug);
} else {
  var detail = result.status === 'networkError'
    ? 'Network error: ' + result.networkError
    : 'HTTP ' + result.response.statusCode + ': ' + result.response.body;
  showDialog('Post Note failed.\n\n' + detail, 'Post Note');
}
```

4. Save. On the shortcut's trigger settings, make sure it's available as a
   launcher icon; text-sharing eligibility comes automatically from the
   `note_text` variable's share setting, not from anything on the
   shortcut itself.

### Shortcut B: "Post Photo Note"

1. Create a new shortcut, type **Regular (HTTP)**.
2. Name it "Post Photo Note".
3. **Request → Method**: `POST`. **URL**:
   `https://api.github.com/repos/froggermtp/personal-site` (never actually
   called in practice — see [design decisions](#design-decisions-for-shortcut-b)).
4. **Request Body / Parameters → Request Body Type**: "Parameters
   (form-data)". Add one parameter: key `photo`, type "Single File",
   upload source "File Picker" (this is the fallback for tapping the icon
   directly; sharing an image in works regardless of this setting).
5. **Response Handling**: set both success and failure output to "None" —
   the script below reports its own result, so the default response UI
   would be redundant or, worse, misleading (since the declared request
   above is a placeholder, not the real work).
6. **Scripting → Run before Execution**, paste:

```js
function padNum(n) { return (n < 10 ? '0' : '') + n; }
function buildSlug(now) {
  return '' + now.getFullYear() + padNum(now.getMonth() + 1) + padNum(now.getDate())
    + '-' + padNum(now.getHours()) + padNum(now.getMinutes());
}
function buildIsoLocal(now) {
  var offsetMinutes = -now.getTimezoneOffset();
  var sign = offsetMinutes >= 0 ? '+' : '-';
  var abs = Math.abs(offsetMinutes);
  return '' + now.getFullYear() + '-' + padNum(now.getMonth() + 1) + '-' + padNum(now.getDate())
    + 'T' + padNum(now.getHours()) + ':' + padNum(now.getMinutes()) + ':' + padNum(now.getSeconds())
    + sign + padNum(Math.floor(abs / 60)) + ':' + padNum(abs % 60);
}
function githubHeaders(token) {
  return {
    'Authorization': 'Bearer ' + token,
    'Accept': 'application/vnd.github+json',
    'X-GitHub-Api-Version': '2022-11-28',
    'Content-Type': 'application/json'
  };
}
function describeFailure(result) {
  if (result.status === 'networkError') return 'Network error: ' + result.networkError;
  return 'HTTP ' + result.response.statusCode + ': ' + result.response.body;
}

if (typeof selectedFiles === 'undefined' || selectedFiles.length === 0) {
  showToast('No image received.');
  abort();
}
var file = selectedFiles[0];

var noteText = getVariable('note_text');
var altText = getVariable('note_alt_text');
if (noteText === null || altText === null) {
  showToast('Post Photo Note cancelled.');
  abort();
}

// UNVERIFIED: 'base64' as the readFile() encoding is a best-effort guess.
// Test with a real photo before trusting this — see docs/phone-publishing.md.
var dir = getDirectory();
var imageBase64 = dir.readFile(file.id, 'base64');

var now = new Date();
var slug = buildSlug(now);
var isoLocal = buildIsoLocal(now);
var headers = githubHeaders(getVariable('github_token'));
var apiBase = 'https://api.github.com/repos/froggermtp/personal-site/contents/';

var imagePath = 'src/assets/images/notes/' + slug + '.jpg';
var imageResult = sendHttpRequest(apiBase + imagePath, {
  method: 'PUT',
  headers: headers,
  body: JSON.stringify({ message: 'Add note image ' + slug, content: imageBase64, branch: 'main' })
});

if (imageResult.status !== 'success') {
  showDialog('Photo upload failed.\n\n' + describeFailure(imageResult), 'Post Photo Note');
  abort();
}

var notePath = 'src/views/notes/' + slug + '.md';
var markdown = '---\n'
  + 'date: ' + isoLocal + '\n'
  + 'image: notes/' + slug + '.jpg\n'
  + 'imageAlt: "' + altText.trim().replace(/"/g, '\\"') + '"\n'
  + '---\n' + noteText.trim() + '\n';

var noteResult = sendHttpRequest(apiBase + notePath, {
  method: 'PUT',
  headers: headers,
  body: JSON.stringify({ message: 'Add note ' + slug, content: base64encode(markdown), branch: 'main' })
});

if (noteResult.status === 'success') {
  showToast('Photo note published: ' + slug);
} else {
  showDialog(
    'Image uploaded, but the markdown commit failed.\n\n' + describeFailure(noteResult)
      + '\n\nThe image is already committed at ' + imagePath + ' -- you may need to add the note manually or retry.',
    'Post Photo Note'
  );
}

abort(); // real work is done above; skip the placeholder declared request
```

7. Save.

## Summary of what's verified vs. assumed

**Verified against the app's public docs or its open-source data model:**
- `getVariable`/`setVariable`, `sendHttpRequest` (synchronous, returns
  `{status, response:{statusCode, body, headers, cookies}, networkError}`),
  `base64encode`, `showToast`, `showDialog`, `abort`, `prompt` all exist
  with the signatures used above.
- `selectedFiles` + `getDirectory()` (no args) + `.readFile(id)` is the
  documented mechanism for reading a shared file's content in a script.
- The "Run before Execution" script (JSON field `codeOnPrepare`) runs
  before the declared HTTP request, and variables can be force-resolved
  early by calling `getVariable()` inside it.
- "Allow Receiving Value from Share Dialog" on a Text Input global
  variable (JSON field `isShareText`) is how text-sharing into a shortcut
  works.
- File parameters (Request Body Type "Parameters (form-data)" with a
  "Single File" parameter, JSON `requestBodyType: "form_data"`,
  `parameters[].type: "file"`) are how a shortcut registers as an image
  share target.
- The full JSON import schema (`version`, `compatibilityVersion`,
  `categories`, `shortcuts`, `variables`, `globalCode`, `codeOnPrepare`,
  etc., and their field-level enum values like `executionType: "scripting"`
  vs `"app"`, `requestBodyType`, `parameterType`, `variableType`) — pulled
  directly from the app's Kotlin data classes and a real test-fixture
  export file in its GitHub repo, not guessed.
- The app accepts a plain (non-zip) JSON file as an import, confirmed from
  the importer's source: it tries to parse the file as a zip first and
  falls back to plain JSON on failure.

**Not verified — test on-device:**
- Whether `readFile(fileId, 'base64')`'s second argument is actually
  honored as an encoding name, or whether some other value/mechanism is
  needed to get binary-safe output. This is the one step in Shortcut B
  that could silently corrupt photos if wrong. See
  ["The one real unknown"](#the-one-real-unknown-reading-photo-bytes).
- Whether `abort()` called from a "Run before Execution" script actually
  suppresses the shortcut's declared HTTP request (Shortcut B is built to
  be harmless either way, so this only affects whether you see one
  extra, ignored network call in the app's history — not correctness).

## Posting from a computer

You can also create notes from your Mac or Linux laptop using `make note`. The workflow generates a new note file with the correct slug, frontmatter, and optional image handling — then opens it in your `$EDITOR` for final edits before you commit and push to `main`.

**Text note:**
```sh
make note
```
Opens your editor with a new, empty note file (frontmatter included).

**Photo note:**
```sh
make note IMG=path/to/photo.jpg ALT="Alt text for accessibility"
```
Copies the image to the assets directory, generates the note file with image metadata, and opens it in your editor. As with the phone shortcuts, the image is copied under the `.jpg` extension regardless of its original format.

After editing, commit and push to publish: `git add -A && git commit -m "Add note <slug>" && git push`. The rest of the pipeline is identical — Actions builds, Netlify deploys.

There's also a browser-based alternative: `make compose` (or just double-click `docs/note-composer.html`) opens a local, self-contained page for writing a note and optionally attaching a photo. In Chromium-based browsers (Chrome, Edge) you can pick the repo folder once and it saves the note markdown and photo straight into `src/views/notes/` and `src/assets/images/notes/`; in other browsers it downloads the files instead for you to move into place by hand. Note for Brave users: Brave disables the File System Access API by default, so the page falls back to downloads — to get direct-into-repo saving, open `brave://flags`, search for "File System Access API" (`brave://flags/#file-system-access-api`), set it to Enabled, and relaunch. Until then you can turn off "Ask where to save each file" in `brave://settings/downloads` to avoid a save dialog per file. When the page has direct folder access it also lists your recent notes, so you can edit or delete an existing note right from the page (edits keep the original slug and timestamp; not available in the download fallback). Nothing the page does ever leaves your machine — there's no server, no upload, no network request. Publishing afterward is the same as above: commit and push.
